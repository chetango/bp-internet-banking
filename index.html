Decisiones Arquitectónicas (ADR)

Cada decisión fue evaluada contra al menos una alternativa. Se justifica técnica y normativamente.

☁️ Nube: AWS sobre Azure

Decisión

Amazon Web Services como proveedor de nube principal.

Justificación 1 — Madurez en sector financiero

AWS tiene mayor adopción en banca latinoamericana a 2026. Cognito, Rekognition, Macie, Shield son servicios maduros con certificaciones PCI-DSS y SOC2 propias.

Justificación 2 — Ecosistema integrado

La combinación Cognito + Rekognition + SQS + DynamoDB resuelve los requerimientos sin necesidad de terceros adicionales, reduciendo la superficie de ataque.

vs Azure AD B2C vs Azure Face API

⚛️ SPA: Next.js sobre Angular

Decisión

Next.js 14 con App Router y RSC como framework para la SPA web.

Justificación 1 — SSR y Core Web Vitals

Next.js ofrece Server-Side Rendering nativo que mejora el LCP (Largest Contentful Paint) crítico para banca. Una pantalla de saldo carga en <1s percibido.

Justificación 2 — Ecosystem y talento

React tiene 5x más desarrolladores disponibles que Angular en Colombia. Menor riesgo de bus factor y facilita contratación futura.

Alternativa evaluada: Angular 17

📱 Mobile: Flutter sobre React Native

Decisión

Flutter 3.x como framework multiplataforma para iOS y Android.

Justificación 1 — Rendimiento nativo real

Flutter compila a código nativo ARM. No usa JavaScript bridge como React Native, lo que es crítico para la cámara del Onboarding facial y la biometría.

Justificación 2 — Un único codebase estable

Flutter tiene garantía de Google con roadmap de largo plazo. local_auth plugin para biometría está mantenido oficialmente y soporta FaceID, TouchID y PIN.

Alternativa evaluada: React Native

🔐 OAuth2: Auth Code + PKCE

Decisión

Authorization Code Flow + PKCE para SPA y App Móvil.

Justificación 1 — Seguridad contra intercepción

PKCE garantiza que solo el cliente que inició el flujo puede canjear el authorization code. Un atacante MitM que capture el code no puede obtener tokens sin el code_verifier.

Justificación 2 — Estándar de la industria

RFC 9700 (OAuth 2.1) depreca oficialmente Implicit Flow y Password Grant. PKCE es el único flujo recomendado para clientes públicos según el estándar vigente.

Descartado: Implicit Flow (deprecado) Descartado: Password Grant

🔄 Transferencias: Patrón Saga

Decisión

Saga de orquestación con compensación para garantizar consistencia en transferencias.

Justificación 1 — No existe transacción distribuida

El débito ocurre en el Core y el crédito también, sin 2PC disponible. Saga es el único patrón que permite consistencia eventual con compensación automática sin bloqueos.

Justificación 2 — Trazabilidad regulatoria

Cada paso de la Saga queda persistido en RDS. Si hay disputa, se puede reconstruir exactamente qué pasó, cuándo y el estado de compensación. Requerido por SFC.

Alternativa evaluada: 2PC (descartado por acoplamiento)

📖 Movimientos: Patrón CQRS

Decisión

Command Query Responsibility Segregation con réplica de lectura dedicada.

Justificación 1 — Ratio lectura/escritura asimétrico

Un cliente típico consulta sus movimientos 10-20 veces por cada 1 que los genera. CQRS permite indexar y optimizar el modelo de lectura sin impactar el Core de escritura.

Justificación 2 — Desacoplamiento del Core legado

La réplica de lectura absorbe todo el tráfico de consultas. El Core solo recibe escrituras, reduciendo su carga en ~90%.

📝 Auditoría: Event Sourcing

Decisión

Event Sourcing con DynamoDB en modo append-only para el log de auditoría.

Justificación 1 — Inmutabilidad regulatoria

La Circular SFC 052/2007 exige logs de operaciones que no puedan ser alterados. Event Sourcing por diseño no permite UPDATE ni DELETE — cada acción es un evento nuevo.

Justificación 2 — DynamoDB para escala

DynamoDB escala horizontalmente sin límite. A 10 años de retención con millones de eventos, no hay degradación de rendimiento en escrituras. PostgreSQL no escalaría igual.

🧠 BFF: Backend for Frontend

Decisión

Un BFF específico para SPA Web y otro para App Móvil.

Justificación 1 — Contratos independientes

La SPA necesita datos enriquecidos para un dashboard complejo. La app móvil necesita payloads mínimos para bajo ancho de banda. Un solo API genera acoplamiento y versiones conflictivas.

Justificación 2 — Seguridad diferenciada

El BFF Web maneja cookies httpOnly seguras. El BFF Mobile maneja tokens en memoria + certificado pinning. Separar permite aplicar políticas de seguridad específicas a cada canal.

Marco Normativo y Regulatorio

Elementos regulatorios que aplican a una entidad bancaria en Colombia bajo supervisión de la SFC.

🔒 PCI-DSS v4.0

Aplica: toda transacción que involucre datos de tarjeta.

  • Req. 2: No usar defaults de fábrica en credenciales
  • Req. 6: Desarrollar sistemas seguros (OWASP Top 10)
  • Req. 8: Autenticación fuerte — MFA obligatorio
  • Req. 10: Logs de auditoría inmutables
  • Req. 12: Política de seguridad de la información
Mandatorio

📋 Circular SFC 052/2007

Aplica: canales electrónicos de entidades vigiladas por la Superintendencia Financiera de Colombia.

  • Doble factor de autenticación para canales digitales
  • Cifrado de información en tránsito y reposo
  • Logs de operaciones mínimo 10 años
  • Plan de continuidad del negocio (BCP/DRP)
  • Notificación obligatoria al cliente post-transacción
Mandatorio Colombia

🧾 Ley 1581/2012 — Habeas Data

Aplica: tratamiento de datos personales de clientes colombianos.

  • Consentimiento explícito para procesamiento de datos biométricos (onboarding facial)
  • Derecho a conocer, actualizar y suprimir datos
  • Registro ante SIC de bases de datos con datos sensibles
  • Cifrado de datos biométricos en reposo (AWS KMS)
Mandatorio Colombia

🌐 ISO 27001 / 27005

Aplica: gestión de seguridad de la información.

  • SGSI implementado: políticas, controles y métricas
  • Gestión de riesgos tecnológicos documentada
  • Pruebas de penetración periódicas (mín. anual)
  • Gestión de vulnerabilidades y parches
Best Practice

💳 FATF / GAFI — AML/KYC

Aplica: prevención lavado de activos y financiamiento terrorismo.

  • Onboarding facial como mecanismo de KYC digital
  • Límites de transferencia configurables por nivel de riesgo
  • Registro de operaciones inusuales (UIAF)
  • Monitoreo de patrones transaccionales
Mandatorio

🔑 OWASP Top 10 — 2025

Aplica: seguridad de aplicaciones web y APIs.

  • A01 Broken Access Control → RBAC en Cognito + API GW
  • A02 Cryptographic Failures → TLS 1.3 + KMS en reposo
  • A03 Injection → ORMs + prepared statements
  • A05 Security Misconfiguration → IaC (Terraform) revisado
  • A07 Auth Failures → MFA + rate limiting + lockout
  • A09 Security Logging → Event Sourcing + CloudWatch
