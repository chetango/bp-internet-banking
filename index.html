Decisiones Arquitectónicas (ADR)

Cada decisión fue evaluada contra al menos una alternativa. Se justifica técnica y normativamente.

☁️ Nube: AWS sobre Azure

Decisión

Amazon Web Services como proveedor de nube principal.

Justificación 1 — Madurez en sector financiero

AWS tiene mayor adopción en banca latinoamericana a 2026. Cognito, Rekognition, Macie, Shield son servicios maduros con certificaciones PCI-DSS y SOC2 propias.

Justificación 2 — Ecosistema integrado

La combinación Cognito + Rekognition + SQS + DynamoDB resuelve los requerimientos sin necesidad de terceros adicionales, reduciendo la superficie de ataque.

vs Azure AD B2C vs Azure Face API

⚛️ SPA: Next.js sobre Angular

Decisión

Next.js 14 con App Router y RSC como framework para la SPA web.

Justificación 1 — SSR y Core Web Vitals

Next.js ofrece Server-Side Rendering nativo que mejora el LCP (Largest Contentful Paint) crítico para banca. Una pantalla de saldo carga en <1s percibido.

Justificación 2 — Ecosystem y talento

React tiene 5x más desarrolladores disponibles que Angular en Colombia. Menor riesgo de bus factor y facilita contratación futura.

Alternativa evaluada: Angular 17

📱 Mobile: Flutter sobre React Native

Decisión

Flutter 3.x como framework multiplataforma para iOS y Android.

Justificación 1 — Rendimiento nativo real

Flutter compila a código nativo ARM. No usa JavaScript bridge como React Native, lo que es crítico para la cámara del Onboarding facial y la biometría.

Justificación 2 — Un único codebase estable

Flutter tiene garantía de Google con roadmap de largo plazo. local_auth plugin para biometría está mantenido oficialmente y soporta FaceID, TouchID y PIN.

Alternativa evaluada: React Native

🔐 OAuth2: Auth Code + PKCE

Decisión

Authorization Code Flow + PKCE para SPA y App Móvil.

Justificación 1 — Seguridad contra intercepción

PKCE garantiza que solo el cliente que inició el flujo puede canjear el authorization code. Un atacante MitM que capture el code no puede obtener tokens sin el code_verifier.

Justificación 2 — Estándar de la industria

RFC 9700 (OAuth 2.1) depreca oficialmente Implicit Flow y Password Grant. PKCE es el único flujo recomendado para clientes públicos según el estándar vigente.

Descartado: Implicit Flow (deprecado) Descartado: Password Grant

🔄 Transferencias: Patrón Saga

Decisión

Saga de orquestación con compensación para garantizar consistencia en transferencias.

Justificación 1 — No existe transacción distribuida

El débito ocurre en el Core y el crédito también, sin 2PC disponible. Saga es el único patrón que permite consistencia eventual con compensación automática sin bloqueos.

Justificación 2 — Trazabilidad regulatoria

Cada paso de la Saga queda persistido en RDS. Si hay disputa, se puede reconstruir exactamente qué pasó, cuándo y el estado de compensación. Requerido por SFC.

Alternativa evaluada: 2PC (descartado por acoplamiento)

📖 Movimientos: Patrón CQRS

Decisión

Command Query Responsibility Segregation con réplica de lectura dedicada.

Justificación 1 — Ratio lectura/escritura asimétrico

Un cliente típico consulta sus movimientos 10-20 veces por cada 1 que los genera. CQRS permite indexar y optimizar el modelo de lectura sin impactar el Core de escritura.

Justificación 2 — Desacoplamiento del Core legado

La réplica de lectura absorbe todo el tráfico de consultas. El Core solo recibe escrituras, reduciendo su carga en ~90%.

📝 Auditoría: Event Sourcing

Decisión

Event Sourcing con DynamoDB en modo append-only para el log de auditoría.

Justificación 1 — Inmutabilidad regulatoria

La Circular SFC 052/2007 exige logs de operaciones que no puedan ser alterados. Event Sourcing por diseño no permite UPDATE ni DELETE — cada acción es un evento nuevo.

Justificación 2 — DynamoDB para escala

DynamoDB escala horizontalmente sin límite. A 10 años de retención con millones de eventos, no hay degradación de rendimiento en escrituras. PostgreSQL no escalaría igual.

🧠 BFF: Backend for Frontend

Decisión

Un BFF específico para SPA Web y otro para App Móvil.

Justificación 1 — Contratos independientes

La SPA necesita datos enriquecidos para un dashboard complejo. La app móvil necesita payloads mínimos para bajo ancho de banda. Un solo API genera acoplamiento y versiones conflictivas.

Justificación 2 — Seguridad diferenciada

El BFF Web maneja cookies httpOnly seguras. El BFF Mobile maneja tokens en memoria + certificado pinning. Separar permite aplicar políticas de seguridad específicas a cada canal.

⚡ Resiliencia: Circuit Breaker con Resilience4j

Decisión

Patrón Circuit Breaker implementado con Resilience4j en todos los microservicios Spring Boot que llaman al Core Bancario o sistemas externos.

Justificación 1 — Protección del Core legado

El Core Bancario es un sistema legado con SLA propio de ~99.5%. Sin circuit breaker, un timeout en el Core provoca cascada de fallos en Customer Service, Account Service y Movements Service. El breaker abre tras 5 fallos consecutivos y retorna respuesta degradada (datos del caché Redis) durante la ventana de 30 segundos, evitando la cascada.

Justificación 2 — Observabilidad del estado

Resilience4j expone métricas de estado (CLOSED/OPEN/HALF_OPEN) a CloudWatch. Se generan alarmas automáticas cuando el breaker abre, permitiendo al equipo de operaciones reaccionar proactivamente antes de que el cliente lo perciba.

Alternativa evaluada: Istio (descartado por complejidad operacional en ECS) Resilience4j vs Hystrix (deprecated)

🔢 API Versioning: URI Path Versioning (/v1/)

Decisión

Versioning por path URI (/v1/movements, /v2/movements) gestionado en el API Gateway. Política de deprecación con 6 meses de notice.

Justificación 1 — Visibilidad y depurabilidad

Las versiones en el URI son visibles en logs, dashboards de CloudWatch y trazas de X-Ray sin configuración adicional. Header versioning (Accept: application/vnd.bp.v2+json) es más purista pero introduce complejidad en debugging y caching de CloudFront.

Justificación 2 — Compatibilidad con clientes móviles

Las apps móviles (Flutter) en producción no se actualizan instantáneamente. La coexistencia de /v1/ y /v2/ en el API Gateway permite migración gradual sin forzar actualizaciones. El Gateway reescribe rutas internas, los microservicios no exponen versiones.

Alternativa evaluada: Header Versioning Alternativa evaluada: Query param (?version=2)

🔑 Secrets Management: AWS Secrets Manager + Rotación Automática

Decisión

Todos los secretos (credenciales de BD, API keys de terceros, client_secret de Cognito) se almacenan en AWS Secrets Manager con rotación automática. Ningún secreto en variables de entorno ni en código fuente.

Justificación 1 — Rotación sin downtime

Secrets Manager rota credenciales RDS automáticamente cada 30 días usando Lambda rotation function. El servicio en ECS recibe el secreto nuevo sin reinicio gracias al SDK que hace GetSecretValue en cada conexión. Rotación manual es propensa a error humano y genera downtime.

Justificación 2 — Cumplimiento PCI-DSS Req. 3.5

PCI-DSS v4.0 exige protección de claves criptográficas y credenciales de acceso. Secrets Manager con KMS CMK (Customer Managed Key) cumple el requisito de encriptación de secretos en reposo y el audit trail de accesos a través de CloudTrail.

Prohibido: secretos en .env o repositorio Alternativa evaluada: SSM Parameter Store (descartado por falta de rotación nativa)

📊 Observabilidad: SLOs y Golden Signals

Decisión

Estrategia de observabilidad basada en los 4 Golden Signals (Latencia, Tráfico, Errores, Saturación) con SLOs definidos por servicio y SLAs comprometidos con negocio.

SLOs definidos por servicio
  • Auth Service: p99 latencia < 300ms · Error rate < 0.1%
  • Movements Service (read): p99 < 200ms · Availability 99.9%
  • Transfer Service: p99 < 2s (incluye Core) · Error rate < 0.05%
  • Onboarding Service: p99 < 5s (incluye Rekognition) · Liveness success > 95%
  • API Gateway: p99 < 100ms overhead · Availability 99.95%
Justificación — Alertas proactivas vs reactivas

Sin SLOs definidos, el equipo de operaciones reacciona a incidentes. Con SLOs y error budgets, se detecta degradación antes de impactar al cliente. CloudWatch Composite Alarms agrega múltiples señales. X-Ray Service Map muestra el grafo de dependencias con latencias en tiempo real.

📊 Comparativa: Fargate vs EC2 para Microservicios

El ejercicio eligió ECS Fargate sobre EC2 + Auto Scaling Groups.

  • Fargate: ~$420/mes. Sin gestión de instancias, parches de SO ni capacity planning. Costo por tarea (vCPU + RAM). Escala a 0 en entornos no-prod.
  • EC2 t4g.small (9 nodos): ~$280/mes. Menor costo pero requiere gestión del clúster, sobreaprovisionamiento para HA, y parches manuales.
  • Decisión: Fargate. El sobrecosto de ~$140/mes se justifica por eliminación de overhead operativo de gestión de EC2, crítico en un equipo pequeño de DevOps.
Elegido: Fargate Alternativa: EC2 ASG

📉 Estrategias de Optimización Implementadas

  • Savings Plans Compute 1 año: cubre Fargate y Lambda automáticamente, −20% sobre on-demand.
  • Reserved Instances RDS 1 año: −40% sobre on-demand para la base de datos principal.
  • Cache-Aside Redis: reduce llamadas al Core Bancario (externo) y a RDS. Ahorro estimado en latencia y costos de integración.
  • DynamoDB On-Demand + TTL + Glacier: paga solo por lo que usa. Archivado automático a $0.004/GB-mes vs $0.25/GB-mes en DynamoDB.
  • HTTP API vs REST API Gateway: −71% en costo de API calls para el mismo volumen.
  • CloudFront cache hit >90%: reduce tráfico de salida desde ECS hacia Internet (egress ~$0.09/GB).
  • X-Ray sampling 5%: reduce costo de trazas en prod sin perder visibilidad estadística.
Ahorro estimado: ~$278/mes

🔔 Controles de Gasto (FinOps)

  • AWS Budgets: alerta al 80% y 100% del presupuesto mensual ($1 400 límite). Acción automática: bloquear lanzamiento de nuevos recursos si se supera 110%.
  • Cost Allocation Tags: todos los recursos etiquetados con env, service, team para visibilidad por componente.
  • AWS Cost Explorer: revisión semanal de anomalías. Anomaly Detection activado con umbral $50/día.
  • Entornos efímeros: ambientes de staging y QA se levantan on-demand y se destruyen fuera de horario laboral. Saving estimado: −60% del costo de no-prod.
  • Rightsizing periódico: revisión trimestral de Compute Optimizer para ajustar tamaño de tareas Fargate y nodos RDS.
